# 系统架构设计论文指南

## 论文考什么

论文不是技术名词说明书，也不是项目工作总结。它要求考生围绕给定架构专题，证明自己能在真实约束下完成架构分析、设计、实施和评价，并能解释决策的得失。

训练目标：读题后 10～15 分钟内完成可写的结构，正文每一节都能回答“为什么、怎么做、结果如何”。实际字数以考试页面要求为准；日常可按摘要约 300 字、正文约 2000～2500 字训练。

## 推荐结构

### 1. 摘要

用一段话交代：

- 项目背景与业务目标；
- 本人角色和主要职责；
- 题目要求对应的核心架构决策；
- 实施效果和结论。

摘要不能只写背景，也不能引入正文没有展开的技术。

### 2. 项目背景

控制篇幅，写清行业、用户、核心流程、规模、约束、原系统痛点和本人职责。背景中的每个数据都应服务于后文决策。

### 3. 方法与概念

用较短篇幅回答题目中的理论问题：定义、步骤、分类或评价方法。不要把教程内容抄成全文主体。

### 4. 方案与实施

这是得分核心。按 3～4 个架构决策展开，每个决策使用同一逻辑：

> 约束/问题 → 备选方案 → 选择理由 → 结构与机制 → 实施难点 → 验证结果

### 5. 评价与改进

写明测试、监控或业务数据；同时诚实说明代价、遗留问题和后续改进。只有优点没有权衡的架构叙述通常缺乏可信度。

## 项目事实卡

每个论文主项目先填写：

| 字段 | 内容要求 |
| --- | --- |
| 项目 | 名称可脱敏，但领域和业务流程要具体 |
| 时间 | 建设或改造周期 |
| 角色 | 本人负责的架构范围和决策边界 |
| 规模 | 用户、请求、数据、节点或团队规模，注明统计口径 |
| 质量目标 | 延迟、吞吐、可用性、RTO/RPO、安全或可修改性 |
| 约束 | 预算、存量、交付期、合规、团队技术能力 |
| 决策 | 至少三个真正影响结构或质量属性的决策 |
| 替代方案 | 至少说明一个没选方案及原因 |
| 验证 | 测试、监控、演练、故障或业务结果 |
| 反思 | 一个真实代价和一个后续改进 |

## 可准备的主题簇

不押单题，按主题簇准备可复用的真实素材：

1. 架构风格选择与应用；
2. 面向质量属性的架构设计；
3. 架构评估、风险、敏感点和权衡点；
4. 微服务、SOA 与服务治理；
5. 事件驱动、消息与最终一致性；
6. 云原生、容器化、弹性与可观测；
7. 高可用、可靠性、容灾与恢复；
8. 安全架构与零信任/纵深防御；
9. 数据架构、缓存、分库分表与数据治理；
10. 软件产品线、领域驱动设计与复用；
11. 遗留系统现代化和架构演化；
12. 性能工程、容量规划与架构验证。

## 写作中的技术深度

以“引入消息队列”为例，最低限度应说明：

- 哪一段同步链路需要解耦，原因是什么；
- 消息模型、主题/队列划分和路由依据；
- 如何处理至少一次投递导致的重复；
- 顺序、一致性、积压和死信如何处理；
- 如何监控端到端延迟和消费失败；
- 该方案牺牲了什么，以及为什么可以接受。

类似地，“采用微服务”需要写清服务边界、数据所有权、通信、事务、治理、部署和团队边界，而不只是列出注册中心等组件。

## 时间分配参考

| 阶段 | 建议时长 | 产出 |
| --- | --- | --- |
| 审题与选择 | 5 分钟 | 选最贴合真实项目的一题，划出所有小问 |
| 提纲 | 10～15 分钟 | 标题、摘要要点、3～4 个核心决策、结果 |
| 正文 | 80～90 分钟 | 完成全部小问，决策链完整 |
| 检查 | 10～15 分钟 | 题目匹配、术语、前后一致、错字和遗漏 |

## 自检评分表

- [ ] 摘要覆盖背景、角色、主题、措施和结果
- [ ] 正文逐一回答题目要求，没有漏问
- [ ] 项目背景具体但不过长
- [ ] 本人职责明确，不把团队工作全写成个人工作
- [ ] 至少三个架构决策有场景依据
- [ ] 每个核心决策解释机制，而非只列产品名
- [ ] 写出替代方案或关键权衡
- [ ] 有可解释的验证指标或事实
- [ ] 术语前后一致，图景没有自相矛盾
- [ ] 结尾包含不足和改进

## 禁忌

- 背诵万能范文，题目换一个关键词仍原文照搬；
- 虚构夸张规模和过度精确的数据；
- 把“我参与”写成“我独自设计并实施全部系统”；
- 全篇解释概念，项目应用不足；
- 全篇写产品配置，缺少架构层面的决策；
- 只有成功，没有问题、代价和复盘。
